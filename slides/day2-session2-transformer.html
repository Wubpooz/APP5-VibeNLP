  • 8 authors from Google Brain & Google Research
  • Originally designed for machine translation

The title is bold and provocative: "Attention Is All You Need." At the time, everyone used RNNs (LSTMs, GRUs) for sequence modeling. The idea of removing recurrence entirely was radical. This is the full Transformer architecture from the paper. The encoder takes the input sequence and builds rich representations. The decoder generates the output sequence one token at a time. Positional Encoding

Teaching the Transformer about word order

Here is a subtle but critical problem. Attention is permutation-invariant: if you shuffle the input tokens, the attention weights change but the mechanism itself has no concept of order. We need to explicitly add position information.

Why Do We Need Position Info?

Self-attention is order-agnostic:

"The cat sat on the mat"

vs.

"The mat sat on the cat"

Without positional encoding, the Transformer sees identical bag-of-tokens for both!

RNNs get order for free (sequential processing).
Transformers must explicitly inject position information.

This is the trade-off of parallelization: we lose inherent ordering. "The cat sat on the mat" and "The mat sat on the cat" would produce the same attention patterns without positional information. We need to add position information to the embeddings so the model knows that "cat" is at position 2 and "mat" is at position 6.

Sinusoidal Positional Encoding

PE(pos, 2i) = sin(pos / 100002i/d)
PE(pos, 2i+1) = cos(pos / 100002i/d)
pos = position index of the word (0, 1, 2 …)
i  = dimension index (0, 1, … d/2−1)
d  = total embedding size (e.g. 512)
Even dims → sin  |  Odd dims → cos

Example: "I love NLP", simplified to d=4

Word pos dim 0
sin(pos/1)		 dim 1
cos(pos/1)		 dim 2
sin(pos/100)		 dim 3
cos(pos/100)
I 0 0.00 1.00 0.00 1.00
love 1 0.84 0.54 0.01 1.00
NLP 2 0.91 −0.42 0.02 1.00

■ dim 0: 0.00 → 0.84 → 0.91  (big jumps per step) → separates adjacent words  |  ■ dim 2: 0.00 → 0.01 → 0.02 (tiny steps) → only differs at pos=50, 100… → separates distant positions

The intuition: think of it like a clock. The seconds hand rotates fast (high frequency), the minutes hand slower, the hours hand even slower. Each moment in time has a unique combination of all these "hands." Similarly, each position in the sequence gets a unique combination of sines and cosines at different frequencies. pos = position in the sequence, i = dimension index, d = model dimension. Low-order dimensions oscillate quickly (capture fine-grained position), high-order dimensions oscillate slowly (capture coarse position relationships). Key property: PE(pos+k) can be expressed as a linear function of PE(pos), so the model can easily learn relative positions.

Fixed vs. Learned Positions

Sinusoidal (Fixed)

  • Generalizes to longer sequences than seen in training
  • No extra parameters to learn
  • Relative positions encoded by linear transformation
  • Used in: original Transformer

Learned Embeddings

  • Each position gets a trainable vector
  • Slightly better performance in practice
  • Limited to max sequence length in training
  • Used in: BERT, GPT

Modern models (2023+): RoPE (Rotary Position Embedding) combines benefits of both

The original paper found that sinusoidal and learned encodings performed nearly identically. In practice, most modern models use learned positional embeddings because they are simple and give marginally better results. The big downside of learned embeddings: if you train with max 512 tokens, you cannot handle 1000-token inputs at inference time. RoPE, used in LLaMA and many recent models, encodes relative positions through rotation matrices. It is now the dominant approach in decoder-only LLMs. This is an active area of research. ALiBi is another alternative.

Add & Norm

The glue that holds each block together

Every sub-layer in the Transformer — whether attention or feed-forward — is wrapped in the same two operations: a residual connection and layer normalization. These are not glamorous, but without them, training a deep Transformer would be practically impossible. Let's understand why.

Add & Norm: Keeping Training Stable

Residual ("Add")

Don't replace — add new info on top of the original.

output = x + Sublayer(x)
token "cat" → self-attention:
x = [1.0, 2.0, 3.0]
Attn = [0.2, 0.5, 0.3]
out = [1.2, 2.5, 3.3]  ✓ original kept

Gradient shortcut → 6+ layers trainable

Layer Norm ("Norm")

After each layer values can explode. Norm resets the scale.

LN(x) = γ · (x − μ) / σ + β
"cat" hidden state (4 dims):
[50, 200, 10, 500]  ← exploding
μ=190, σ≈186
[−0.75, 0.05, −0.97, 1.67]  ✓ stable

Per-token → works with variable-length sequences

Combined: output = LayerNorm(x + Sublayer(x))

Applied twice per encoder layer: after self-attention, and after FFN

Residual connections come from ResNet (He et al., 2015). The key intuition: instead of learning a full transformation H(x), the layer only needs to learn the "residual" F(x) = H(x) - x. If a layer is useless, it can learn F(x)=0 and pass x unchanged. Critically, gradients flow back through the identity shortcut without vanishing — this is why 6-layer (and deeper) Transformers are trainable. Example: x=[1,2,3], Attn(x)=[0.2,0.5,0.3], output=[1.2,2.5,3.3]. The original token meaning is preserved and enriched, not replaced. Layer normalization (Ba et al., 2016) normalizes across the feature dimension for each token independently. For [50,200,10,500]: mean=190, std≈186, normalized ≈ [-0.75, 0.05, -0.97, 1.67]. γ and β let the model rescale if useful. Why layer norm instead of batch norm? Variable-length sequences make batch statistics unreliable. Layer norm works per-token, independent of batch. Note: original paper = Post-LN (after residual). Modern models = Pre-LN (before sublayer) — trains more stably at large scale.

Feed-Forward Networks

The "thinking" layer in each block

The FFN is a simple two-layer neural network with a ReLU activation in between. "Position-wise" means it is applied to each token independently -- there is no interaction between tokens here. That is the job of attention. The expansion to 4x is important: it gives the model a larger "workspace" to transform the representation. Think of it as a bottleneck architecture. Recent work suggests that FFN layers act as "key-value memories" that store factual knowledge (Geva et al., 2021). This is why larger FFN dimensions correlate with more knowledge capacity. Modern models often use GELU instead of ReLU, and some use gated variants (SwiGLU).

Walk through this slowly. Let's trace a concrete token, say "cat", through the block. Step 1: "cat" gets its embedding vector (learned) plus its positional encoding. Step 2: This vector is projected into Q, K, V. All tokens do the same. Multi-head attention computes 8 parallel attention operations. Step 3: We add the original input (residual) to the attention output, then normalize. Step 4: The FFN independently transforms each token's representation. Step 5: Another residual + normalization. This output goes to the next layer. After 6 layers, we have rich contextual representations. Ask students: "Why do we have two Add&Norm steps?" (One for attention, one for FFN)

How the Decoder Generates Text

The decoder generates ONE TOKEN AT A TIME, step by step

The autoregressive loop:

[START]
→
"The"
→
"The cat"
→
"The cat sat"
→
"The cat sat on"
→
[END]

At each step, the decoder sees:

  • All previously generated tokens (its own output so far)
  • The full encoder output (source representation, via cross-attention)

Why masked attention exists:

During training we process all positions at once for efficiency. But we must mask future tokens to simulate the step-by-step generation the model will do at inference time.

The mask makes training parallel while preserving the left-to-right property

This slide is the conceptual bridge before we look at the decoder's architecture. Inference is inherently sequential: generate one token, feed it back as input, repeat. This is "autoregressive" generation. During TRAINING though, we have the full target sequence available. We process ALL positions in parallel — but only if we prevent the model from cheating by looking at future tokens. The causal mask (upper triangular -infinity matrix) achieves exactly this: position t can only attend to positions 1 through t. Training is parallel, but the model learns the correct left-to-right conditional distribution. This is why the decoder's first attention layer is called "Masked Self-Attention."

Decoder: What's Different?

The decoder has three key differences from the encoder:

1 Masked Self-Attention: Each position can only attend to earlier positions. Prevents "seeing the future" during generation.
2 Cross-Attention: Queries from decoder, Keys & Values from encoder output. This is how the decoder "reads" the input.
3 Autoregressive generation: Output is generated one token at a time, left to right.

The mask in masked self-attention is an upper-triangular matrix of negative infinity values added before softmax, forcing future positions to have zero attention weight. Cross-attention is the bridge between encoder and decoder. The decoder's query says "what am I looking for?" and the encoder's keys and values provide the context. This is analogous to how in Session 1 we described attention in sequence-to-sequence models: the decoder attends to the encoder states. During training, all positions are computed in parallel using teacher forcing (we feed the ground truth). During inference, generation is sequential.

Here is the complete picture. Input goes through embedding + positional encoding, then through 6 encoder layers, each with self-attention and FFN. The encoder output feeds into the decoder's cross-attention. The decoder also has 6 layers: masked self-attention, cross-attention, FFN. Finally, a linear layer + softmax produces probabilities over the output vocabulary. The original model had about 65 million parameters. Compare to GPT-3 with 175 BILLION parameters -- same basic architecture, just scaled up. The architecture is elegant: it is built entirely from attention, FFN, and normalization. No recurrence, no convolutions.

Let's Think About It

Discussion Question

In the encoder, self-attention lets every token see every other token.
In the decoder, masked attention only lets tokens see past tokens.

Why is this masking necessary in the decoder?
What would happen without it?

Hint: Think about how the model is used at inference time...

Give students 2-3 minutes to think/discuss. Answer: During inference, the decoder generates one token at a time. When generating token t, tokens t+1, t+2, ... do not exist yet. If we allowed the model to see future tokens during training, it would learn to "cheat" by just copying the answer. At inference time, those future tokens would not be available, and the model would fail. The mask ensures training conditions match inference conditions. This is the key difference between encoder (bidirectional) and decoder (unidirectional).

Transformer Variants

Encoder-only, Decoder-only, Encoder-Decoder

The original Transformer used both encoder and decoder. But researchers quickly discovered that you could use just one side for different types of tasks. This led to three major families.

Three Transformer Families

Encoder-Only

Self-Attention (bidirectional)

BERT (2018)

Sees entire input at once

Best for: classification, NER, QA extraction

Decoder-Only

Masked Self-Attention (causal)

GPT (2018–)

Sees only past tokens

Best for: text generation, chat, code

Encoder-Decoder

Full Architecture

T5 (2019)

Encoder reads, decoder generates

Best for: translation, summarization

Encoder-only (BERT): Uses only the encoder with bidirectional attention. Every token sees every other token. Great for understanding tasks. Trained with masked language modeling (predict hidden words). Decoder-only (GPT): Uses only the decoder with causal masking. Each token only sees previous tokens. Great for generation tasks. Trained with next-token prediction. This is what powers ChatGPT, Claude, and most modern LLMs. Encoder-Decoder (T5, BART): Uses the full architecture. Best for tasks that map one sequence to another. T5's insight: frame every NLP task as text-to-text. The big trend: decoder-only models have dominated since GPT-3 showed that scale + generation = powerful general intelligence.

Why Transformers Won

The end of the RNN era

Let's step back and understand why this architecture didn't just work but completely dominated everything before it.

Why Transformers Won

Property RNN / LSTM Transformer
Parallelization Sequential (O(n) steps) Fully parallel (O(1) steps)
Long-range dependencies Degrades over distance Direct connection (any distance)
Training speed Slow (cannot use GPUs fully) Fast (GPU-friendly matrix ops)
Scalability Diminishing returns at scale Performance grows with scale
Pretraining Limited transfer learning Excellent pretrain → finetune

The scaling hypothesis: Transformer performance improves predictably with more data, compute, and parameters

Parallelization is the most immediate advantage. RNNs must process token by token. Transformers process all tokens simultaneously with matrix multiplication. This maps perfectly to GPUs which are designed for parallel matrix operations. Long-range dependencies: In an RNN, information from token 1 must pass through every intermediate hidden state to reach token 100. In a Transformer, token 1 directly attends to token 100 in a single operation. The scaling story is perhaps most important: OpenAI discovered that Transformer performance scales as a smooth power law with data, parameters, and compute. This "scaling law" is why we went from 65M to 175B+ parameters. RNNs do not exhibit these clean scaling properties. Transfer learning: pretraining on massive text then fine-tuning on specific tasks works remarkably well with Transformers. This is the paradigm shift.

Beyond Autoregressive

The Next Paradigm Shift?

A new challenger: Diffusion LLMs

This section is OPTIONAL — only cover if time permits. Everything we've discussed so far — GPT, BERT, all modern LLMs — follows the autoregressive paradigm: generate one token at a time, left to right. But a new approach is emerging that could fundamentally change how language models work.

The Autoregressive Bottleneck

Current LLMs: one token at a time, sequentially

The
→
cat
→
sat
→
on
→
the
→
mat

Problem: Generation speed is fundamentally limited

  • Must wait for each token before generating the next
  • GPU parallelism underutilized during decoding
  • Latency compounds in agentic loops (10s of LLM calls)

Like a typewriter: one letter at a time, no matter how fast you type.

This is a fundamental architectural constraint, not just an engineering problem. Even with the fastest GPUs, autoregressive models are bottlenecked by sequential generation. In agentic workflows where you might have 20-50 LLM calls chained together, this latency compounds dramatically.

Diffusion LLMs: Parallel Generation

A fundamentally different approach: generate all tokens simultaneously

Autoregressive

Like a typewriter

  • One token at a time
  • Left-to-right, sequential
  • ~50-200 tokens/sec typical

Diffusion

Like an editor revising a draft

  • All tokens generated at once
  • Iteratively refined in parallel
  • 1000+ tokens/sec achieved

Mercury 2 (Inception Labs, 2026): 1,009 tokens/sec on NVIDIA Blackwell
"The fastest reasoning LLM, powered by diffusion"

Diffusion models work by starting with noise and iteratively refining toward coherent output. You've seen this in image generation (Stable Diffusion, DALL-E). Now it's being applied to text: start with random tokens, refine all of them in parallel over a few steps. Mercury 2 from Inception Labs achieved 1009 tokens/sec — more than 5x faster than typical autoregressive models. This is a paradigm shift, not just an optimization.

How Diffusion LLMs Work

Step 1: Start with noisy/random tokens
[###] [???] [~~~] [***] [&&&] [!!!]
Step 2: Parallel refinement — all positions updated simultaneously
[The] [cat] [sit] [in] [a] [mat]
Step 3: Iterate a few times until convergence
[The] [cat] [sat] [on] [the] [mat]

Key insight: Number of steps is constant, not proportional to sequence length
100 tokens or 1000 tokens — same number of diffusion steps

This is the key algorithmic difference. In autoregressive models, generating N tokens requires N forward passes. In diffusion LLMs, generating N tokens requires only K diffusion steps, where K is typically small (10-50) and fixed regardless of N. This is why diffusion can be so much faster: the GPU can parallelize across all token positions at once. The model learns to denoise/refine all positions simultaneously, converging to coherent text.

Why This Matters

Real-time voice AI: sub-200ms responses enable natural conversation
Agentic loops: 50 LLM calls at 1000 tok/s vs 100 tok/s = 10x faster workflows
Code autocomplete: suggestions that feel instant, not lagging
Cost efficiency: same quality at lower inference cost

Open question: Will diffusion LLMs match autoregressive quality at scale?

Mercury 2 claims competitive quality with GPT-5.2 at >5x the speed

Source: Inception Labs — "Introducing Mercury 2" (2026)

The implications are significant. Voice AI has been limited by latency — diffusion could make it truly real-time. Agentic workflows compound latency; faster inference changes what's architecturally possible. The open question is quality at scale. Autoregressive models have proven they scale to frontier capabilities. Diffusion LLMs are newer and less proven at the largest scales, but Mercury 2 claims competitive quality. This is an active research area — by the time you're working in industry, diffusion LLMs may be mainstream.
