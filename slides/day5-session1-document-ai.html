PDF Parsing Techniques

From rule-based to LLM-powered extraction

Now let's get practical. PDF parsing is the foundation of Document AI. The approaches range from simple text extraction to sophisticated layout-aware models. Understanding this spectrum is critical for choosing the right tool for your project.

Each generation builds on the previous -- you often need multiple techniques in one pipeline.

PDF parsing has evolved through four major generations. Rule-based tools like PyPDF extract text from digital PDFs but can't handle scanned documents or complex layouts. OCR tools like Tesseract can read scanned documents but lose structural information. Layout-aware models like LayoutLM understand that position matters -- a number next to "Total" means something different than next to "Date". And now LLMs can directly process document images. In practice, you'll often combine multiple techniques.

Rule-Based Extraction

PyPDF2 / pypdf

# Simple text extraction
from pypdf import PdfReader

reader = PdfReader("invoice.pdf")
for page in reader.pages:
    text = page.extract_text()
    print(text)

Good for: digital-born PDFs with simple layouts

pdfplumber

# Table-aware extraction
import pdfplumber

with pdfplumber.open("invoice.pdf") as pdf:
    page = pdf.pages[0]
    # Extract tables as lists
    tables = page.extract_tables()
    # Get text with position info
    words = page.extract_words()

Good for: tables, forms with consistent structure

Limitation: cannot handle scanned documents, complex multi-column layouts, or handwriting

Let's start with the simplest tools. PyPDF extracts raw text from digital PDFs -- those created by software, not scanned. pdfplumber goes further: it can extract tables by detecting cell boundaries and gives you word-level position information. These tools are fast, lightweight, and perfect for simple documents. But they fail completely on scanned documents, handwriting, or complex layouts. Use them as your first line of attack, then fall back to more sophisticated methods when needed.

OCR-Based Extraction

Tesseract OCR

import pytesseract
from PIL import Image

# Basic OCR
text = pytesseract.image_to_string(
    Image.open("scan.png"),
    lang='eng+fra'  # multilingual
)

# With bounding boxes
data = pytesseract.image_to_data(
    Image.open("scan.png"),
    output_type=pytesseract.Output.DICT
)

Open source, 100+ languages, Google-maintained

PaddleOCR

from paddleocr import PaddleOCR

ocr = PaddleOCR(
    use_angle_cls=True,
    lang='en'
)

result = ocr.ocr("document.jpg")
for line in result[0]:
    bbox, (text, conf) = line
    print(f"{text} ({conf:.2f})")

Fast, accurate for CJK, angle detection built-in

Key advance: OCR gives you text + position, but no semantic understanding of document structure

When documents are scanned images, you need OCR -- Optical Character Recognition. Tesseract is the gold standard open-source OCR engine, maintained by Google, supporting over 100 languages. PaddleOCR from Baidu is particularly strong for Chinese, Japanese, and Korean text, and it includes angle detection which handles rotated text. Both give you text plus bounding box positions. The key limitation is that OCR extracts characters but doesn't understand what they mean or how the document is structured. It sees "1,500.00" and "Total Due" as separate text blocks without knowing they're related.

Microsoft Research • Pre-trained on IIT-CDIP (11M document images) • Fine-tune for your task

This is where things get really interesting. LayoutLM from Microsoft Research was a breakthrough. The key insight: position matters in documents. A number in the top-right corner of an invoice is probably the invoice number. The same number at the bottom is probably the total. LayoutLM combines three types of information: text tokens from OCR, 2D position embeddings that encode where text appears on the page, and visual features from the document image. It's pre-trained on 11 million documents and can be fine-tuned for specific tasks. LayoutLMv2 added deeper image integration, and v3 unified all three modalities from the start.

Modern Approach: LLM-Based Extraction

from anthropic import Anthropic
import base64

client = Anthropic()

# Read the PDF/image
with open("invoice.pdf", "rb") as f:
    data = base64.standard_b64encode(f.read()).decode()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": [
            {"type": "image", "source": {"type": "base64",
                "media_type": "application/pdf", "data": data}},
            {"type": "text", "text": """Extract these fields as JSON:
            - invoice_number, date, vendor_name
            - line_items: [{description, quantity, unit_price, total}]
            - subtotal, tax, total_due"""}
        ]
    }]
)

No training data needed. No fine-tuning. Works out of the box on any document format.

And here's the modern approach that's changing everything. Instead of building OCR pipelines and fine-tuning LayoutLM, you can send a document image directly to a vision-language model like Claude and ask it to extract structured data. This code sends a PDF to Claude and asks for structured JSON output. No training data. No fine-tuning. No OCR pipeline. It handles complex layouts, multiple languages, handwriting, and even charts. The tradeoff is cost and latency -- processing thousands of documents this way can be expensive. But for many use cases, the simplicity and accuracy make it the best choice.

PDF Parsing: Choosing Your Tool

Approach Best For Accuracy Speed Cost
Rule-based
PyPDF, pdfplumber		 Digital PDFs, simple tables Low-Med Very Fast Free
OCR
Tesseract, PaddleOCR		 Scanned docs, images Medium Medium Free
Layout-aware
LayoutLM, Donut		 Forms, invoices (trained) High Medium GPU needed
LLM-based
GPT-4V, Claude		 Any document, zero-shot Very High Slow $$-$$$

Pro tip: use rule-based for easy docs, LLM for complex ones — hybrid pipelines win.

Here's a decision matrix. Rule-based tools are fast and free but only work on simple digital PDFs. OCR handles scanned documents but doesn't understand structure. Layout-aware models like LayoutLM give high accuracy on trained document types but require GPU and training data. LLM-based extraction works on anything with zero training but is slow and expensive. In practice, the best systems are hybrids: use pdfplumber first, fall back to OCR for scanned pages, and use an LLM for complex or unusual documents. This tiered approach optimizes both cost and accuracy.

Part 3

Document Understanding

Beyond extraction: classification, NER, tables

Now that we can extract text from documents, the next challenge is understanding what the text means. Document understanding encompasses several key tasks that go beyond simple extraction.

Document Understanding Tasks

Key Information Extraction (KIE)
Extract specific fields: invoice number, date, amount, vendor. Structured output from unstructured input.
Tools: LayoutLM + fine-tuning, LLM prompting
Document Classification
Categorize: invoice vs. contract vs. receipt vs. report. Route documents in processing pipelines.
Tools: BERT/LayoutLM classifier, zero-shot LLM
Table Extraction
Detect table boundaries, parse rows/columns, handle merged cells and spanning headers.
Tools: pdfplumber, Camelot, Table Transformer
Document NER
Named Entity Recognition adapted for documents: dates, monetary values, addresses, legal entities.
Tools: spaCy, LayoutLM token classification
Four core tasks. KIE is about extracting specific fields -- "What is the invoice number? What is the total?" Document classification routes documents -- is this a contract or an invoice? Table extraction is notoriously difficult because tables come in infinite formats with merged cells and spanning headers. Document NER adapts the Named Entity Recognition you learned in Day 1 to handle document-specific entities like monetary amounts, dates in various formats, and addresses. Each task has specialized tools and can also be handled by general-purpose LLMs.

# Example: extract fields with structured output
from pydantic import BaseModel
from anthropic import Anthropic

class Invoice(BaseModel):
    invoice_number: str
    date: str
    vendor: str
    line_items: list[dict]
    total: float

# Use LLM with structured output schema
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    messages=[{"role": "user", "content": [
        {"type": "image", "source": {...}},  # document image
        {"type": "text", "text": f"Extract invoice data matching: {Invoice.schema_json()}"}
    ]}]
)
invoice = Invoice.model_validate_json(response.content[0].text)
Here's what a real KIE pipeline looks like. The document goes through OCR or text extraction, layout analysis to understand structure, field extraction to pull out specific values, and validation to ensure the output is correct. The code example shows how to combine an LLM with Pydantic for structured output. You define an Invoice schema, send the document image to Claude, and parse the response into a validated Python object. This pattern -- LLM plus Pydantic -- is extremely powerful for building production document processing systems.

Part 4

Beyond Text: The Multimodal Shift

Why NLP alone is not enough

Now let's zoom out. Document AI naturally leads us to multimodal AI because documents contain more than text -- they have images, charts, tables, and visual layouts. This transition from text-only NLP to multimodal understanding is one of the most important trends in AI right now.

Why Multimodal?

"The world is not just text. Humans understand through sight, sound, and language simultaneously. AI must do the same."
— Yann LeCun, 2023

Text-Only NLP Limits

  • Can't read charts or graphs
  • Misses visual layout cues
  • No understanding of images
  • Ignores spatial relationships

Multimodal Enables

  • Document visual understanding
  • Image + text reasoning
  • Chart and diagram analysis
  • Real-world grounded AI
Why multimodal? Because the real world is multimodal. When you look at a document, you don't just read the text -- you see the layout, the logos, the charts. Text-only NLP literally can't see these things. It can't read a bar chart, understand a photo in a news article, or parse the visual structure of a form. Multimodal AI closes this gap. And as we saw with Document AI, some of the hardest problems -- like understanding complex document layouts -- are fundamentally visual problems.

Vision-Language Models

CLIP, GPT-4V, Claude Vision, Gemini, LLaVA

Now we dive into vision-language models -- the architectures that connect visual understanding with language. We'll start with CLIP, the foundational model, then cover the latest commercial and open-source options.

CLIP: Contrastive Language-Image Pre-training

Trained on 400M image-text pairs from the internet

CLIP is the foundational model that connected vision and language. The architecture is elegant: two separate encoders -- a Vision Transformer for images and a text Transformer for captions. During training, CLIP processes batches of image-text pairs and learns to maximize the similarity between matching pairs while minimizing similarity for non-matching pairs. This is contrastive learning. It was trained on 400 million image-text pairs scraped from the internet. The result is a shared embedding space where images and text with similar meaning end up close together.

CLIP: What Can You Do With It?

Zero-Shot Classification

from transformers import CLIPProcessor, CLIPModel

model = CLIPModel.from_pretrained(
    "openai/clip-vit-base-patch32"
)
processor = CLIPProcessor.from_pretrained(
    "openai/clip-vit-base-patch32"
)

# Classify without training!
labels = ["invoice", "contract",
          "receipt", "letter"]
inputs = processor(
    text=labels,
    images=image,
    return_tensors="pt"
)
outputs = model(**inputs)
probs = outputs.logits_per_image.softmax(dim=-1)

Semantic Image Search

# Index images
image_embeddings = []
for img in document_images:
    inputs = processor(
        images=img, return_tensors="pt")
    emb = model.get_image_features(**inputs)
    image_embeddings.append(emb)

# Search with text query
query = "graph showing revenue growth"
text_inputs = processor(
    text=query, return_tensors="pt")
text_emb = model.get_text_features(**text_inputs)

# Find most similar images
similarities = [
    cosine_sim(text_emb, img_emb)
    for img_emb in image_embeddings]

CLIP's shared embedding space enables text↔image matching without task-specific training

CLIP enables two killer applications. First, zero-shot image classification: you describe categories in natural language, and CLIP matches images to the best description -- no training data needed. The code on the left classifies a document image as invoice, contract, receipt, or letter without any labeled examples. Second, semantic image search: you embed all your images with CLIP's vision encoder, then search using natural language queries. The code on the right finds images matching "graph showing revenue growth" by computing cosine similarity in the shared embedding space. Both of these are directly useful for Document AI.

Liu et al., 2023 • "Visual Instruction Tuning"

Key idea: visual instruction tuning — train on GPT-4-generated visual Q&A data

LLaVA's architecture is beautifully simple. It takes CLIP's vision encoder, adds a projection layer to map image features into the language model's token space, and feeds them alongside text tokens into a language model like Vicuna or LLaMA. The key innovation is visual instruction tuning: they used GPT-4 to generate high-quality visual question-answer pairs, then fine-tuned the model on this data. The result is an open-source model that can understand images and answer questions about them. You can run this on a single GPU -- making it practical for research and privacy-sensitive applications where you can't send data to external APIs.

Multimodal Applications for NLP

VQA, captioning, document VQA, multimodal RAG

Now let's look at the applications that combine vision and language. These are the tasks you can build into your Demo Day projects.

Key Multimodal Applications

Visual Question Answering (VQA)
Ask natural language questions about images. "What color is the car?" "How many people are in the photo?"
Benchmark: VQAv2, OK-VQA
Image Captioning
Generate natural language descriptions of images. Foundation for accessibility, search, and content management.
Models: BLIP-2, CoCa, LLaVA
Document VQA
Answer questions about documents: read charts, parse tables, understand forms. Combines OCR + reasoning.
Benchmark: DocVQA, InfographicVQA, ChartQA
Multimodal RAG
Retrieve and reason over both text and images. Index figures, diagrams, and charts alongside text for complete answers.
Tools: CLIP embeddings + text embeddings in vector DB
Four major application areas. VQA is the classic task -- asking questions about images. Image captioning generates descriptions -- essential for accessibility and search. Document VQA is the intersection of Document AI and multimodal -- answering questions about documents by reading charts, tables, and forms. And multimodal RAG extends the RAG pattern you learned on Day 4 to include images. Instead of just retrieving text chunks, you also retrieve relevant figures, diagrams, and screenshots, and feed them to a VLM for comprehensive answers.

Multimodal RAG Architecture

Text Chunks
↓ embed
Text Embeddings
(e.g., OpenAI, Cohere)

Enables answering questions like: "According to the Q3 report chart, what was the revenue trend?"

Here's how multimodal RAG works. You index both text chunks and images into a vector database. Text gets embedded with a text embedding model, images with CLIP or SigLIP. When a user asks a question, you retrieve both relevant text and relevant images, then pass everything to a VLM that can reason over both modalities. This is incredibly powerful for enterprise applications. Imagine a financial analyst asking about a quarterly report -- the system retrieves the relevant text paragraphs AND the revenue chart, then the VLM synthesizes a complete answer. This is the future of document intelligence.
