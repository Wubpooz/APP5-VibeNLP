The AI Coding Revolution

Context before the presentations

Before the groups present, let's set the stage with a brief overview of how we got here and what the landscape looks like in March 2026.

2021: GitHub Copilot launches — basic autocomplete, one line at a time
2023: ChatGPT changes everything — AI chat panels appear in editors

Walk through the timeline. Key point: in just 5 years, we went from "AI suggests the next line" to "AI builds entire features autonomously." This pace of change is unprecedented in software engineering tooling. The 2025-2026 shift is particularly dramatic — agents now drive, humans review.

This is the key framework for understanding the landscape. Category 1: You stay in your familiar editor, AI is an add-on. Category 2: The editor itself is redesigned with AI as a first-class citizen. Category 3: No GUI at all — you work in the terminal, describe goals, AI executes. Most developers in 2026 use a combination. The spectrum is not a hierarchy.

Cursor

VS Code Fork, Rebuilt Around AI

  • Tab Completion: predictive, multi-line, context-aware
  • Chat: @ mentions for precise context
  • Composer / Agent Mode: multi-file edits with a single prompt
  • Familiar UX: all VS Code extensions work

Context System

@file @folder @web @docs @codebase @git

Tell the AI exactly what to look at

Cursor is probably the most well-known AI-native IDE right now. It's a fork of VS Code, so all your extensions, keybindings, and settings carry over. The @ mention system is one of its best features — you can point the AI at specific files, folders, documentation, or the entire codebase for context. Composer mode is where it gets really powerful — describe a feature and it edits multiple files at once, showing you a diff to approve.

Windsurf

By Codeium — The Agentic Competitor

  • Also a VS Code fork — familiar feel
  • Cascade: multi-step agentic flow — chains actions autonomously
  • Flows: learns and remembers your coding patterns
  • Generous free tier — great for students

Cascade in Action

1. You: "Add user auth to this app"
↓
2. Cascade reads your codebase
↓
3. Creates routes, models, middleware
↓
4. Runs tests, fixes errors
↓

Windsurf is Codeium's answer to Cursor. The key differentiator is Cascade, which feels more "agentic" — it can chain multiple steps without you having to prompt each one individually. Flows is their feature for learning your patterns over time, so suggestions become more personalized the more you use it. The free tier is notably more generous than Cursor's free tier.

Key Differentiators vs Cursor

  • More autonomous — Cascade chains actions without re-prompting
  • Better context awareness — Flows learn your patterns
  • $5/mo cheaper at the Pro tier
  • Slightly less mature ecosystem than Cursor
The price advantage is real — $15 vs $20 per month for Pro. The Cascade feature is genuinely impressive for multi-step tasks. However, Cursor has a larger community and more tutorials available. For students, the free tier is a great way to get started. Honest assessment: both are excellent, the differences are narrowing.

Cline

Open Source — VS Code Extension

  • Not a fork! — installs as a VS Code extension
  • Fully open source on GitHub
  • Bring your own key — works with any LLM provider
  • Reads/writes files, runs terminal commands
  • No vendor lock-in, transparent, customizable

Cost Model

Tool Free & open source
API Pay per use
Example: Claude API at ~$3/$15 per 1M tokens (input/output)
Typical session: $0.10 — $2.00
Cline is the open-source champion in this space. The key difference: it's a VS Code EXTENSION, not a fork. So you keep your exact VS Code installation, with all your settings. The bring-your-own-key model means you can use Claude, GPT-4o, Llama, or any other model. You pay the API provider directly. For students who want to understand how these tools work under the hood, Cline's open source code is a great learning resource. Cost can be unpredictable though — heavy usage days can get expensive.

Claude Code

A Different Approach: Terminal-First, Agent-First

Now let's look at Claude Code separately because it represents a fundamentally different paradigm. It's not an IDE and not an extension — it's a CLI agent.

Claude Code — How It Differs

  • Terminal-first: no GUI, runs in your shell
  • You describe the goal, Claude drives, you review
  • 5.5x fewer tokens than Cursor for same tasks
  • Best for: large refactors, test suites, documentation, understanding codebases

Typical Workflow

$ claude
> Add authentication to this Flask app
  using JWT tokens. Include tests.

Claude reads your codebase...
Claude creates auth/routes.py...
Claude creates auth/models.py...
Claude creates tests/test_auth.py...
Claude runs pytest... all pass ✓

> Looks good, commit it.
Claude Code is the tool students are using in this course. The key mental model: you are the architect, Claude is the builder. You describe what you want at a high level, Claude figures out the implementation. The 5.5x token efficiency comes from Anthropic's internal benchmarks — because the agent plans before acting, it wastes fewer tokens. The terminal-first approach means it works with ANY editor — no lock-in.

How to Choose

A Decision Framework

Budget-conscious student?
→ GitHub Copilot (free via Education) + Cline (free + your API key)
Want the best IDE experience?
→ Cursor Pro ($20/mo) — smoothest UX, largest community
Want agentic power?
→ Claude Code ($20/mo Claude Max) — best for complex, multi-file tasks
Want open source?
→ Cline + Aider + Continue.dev — full stack, no vendor lock-in
Best combo for this course?
→ IDE tool (Cursor or Copilot) + Claude Code = daily coding + big tasks
This is the practical slide. Students need actionable advice. The combo recommendation is key — these tools are complementary, not competing. Use an IDE tool for the moment-to-moment coding experience (completions, quick edits). Use Claude Code when you need to understand a large codebase, do a major refactor, write a test suite, or generate documentation. For the NLP project specifically, Claude Code is excellent for setting up the project structure, writing boilerplate, and generating tests.

Practical Tips for AI-Assisted Coding

  1. Be specific in your prompts
    Bad: "Fix the bug" — Good: "Fix the authentication timeout error in login.py line 42"
  2. Provide context
    Use @file, @docs, or paste error messages — the AI cannot read your mind
  3. Review every change
    AI is fast but not infallible — always read the diff before accepting
  4. Use Git frequently
    Commit before AI edits so you can revert — small commits, clear messages
  5. Learn the shortcuts
    Tab to accept, Cmd+K for inline edits, Cmd+L for chat — speed matters
These tips apply to ALL the tools we've discussed. The Git tip is crucial — students should commit frequently when using AI tools. An AI can make sweeping changes that are hard to undo without Git. The prompt quality tip is the single most impactful advice: the quality of AI output is directly proportional to the quality of your prompt. Encourage students to practice prompt engineering as a skill.

Common Pitfalls

Don't...

  • Blindly accept all suggestions
  • Use AI without understanding the code
  • Paste sensitive data (API keys, passwords)
  • Expect it to work perfectly first time
  • Skip writing tests because "AI wrote it"

Do...

  • Read and understand every line
  • Use AI to learn — ask "explain this code"
  • Set up .gitignore and .env first
  • Iterate: prompt → review → refine
  • Write tests WITH AI, then verify them

The goal: AI amplifies your skills, it doesn't replace understanding

This slide is important for setting the right mindset. Students might be tempted to let AI do everything — but that's a trap. If you don't understand the code, you can't debug it when things break. The security tip is critical — never paste API keys or credentials into AI chat. The test tip: AI-written tests are better than no tests, but verify they actually test what they claim to test. AI loves writing tests that always pass.

Where This Is Heading

"In the near future, every line of code will be written, reviewed, or optimized by AI. The developer's role shifts from typist to architect."

— The emerging consensus, 2026

  • AI tools will converge — every IDE will have agent mode
  • MCP (Model Context Protocol) will standardize tool ↔ model communication
  • The best developers will be the best AI collaborators
This is the forward-looking slide. The tools we discussed today are just the beginning. MCP (Model Context Protocol by Anthropic) is a standard for how AI tools communicate with external systems — think of it as USB for AI. The convergence point: in 2-3 years, the distinction between Cursor, Windsurf, and Copilot may not matter — they'll all offer similar capabilities. What WILL matter is your ability to work effectively with AI.
