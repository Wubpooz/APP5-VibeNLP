MCP

The USB Port for AI

Model Context Protocol — connecting AI to the real world

Now we move to MCP, which is arguably the most important infrastructure concept in modern AI tooling. The USB analogy is deliberate -- just as USB gave us one standard port for all devices, MCP gives AI one standard protocol for all tools.

What is MCP?

Open standard by Anthropic (Nov 2024) for AI-to-tool integration

The LSP Analogy

Remember LSP (Language Server Protocol)? Before LSP, every IDE needed a custom plugin for every language. LSP standardized this: one protocol, all languages.

MCP does the same for AI: one protocol to connect any AI model to any tool.

AI Model
(Claude, GPT, ...)
↔
MCP Protocol
(JSON-RPC)
↔
Tools / APIs / Data
(GitHub, DB, Slack, ...)

Without MCP: every AI tool needs custom integrations for every service (N x M problem)
With MCP: build one server, connect to any AI client

For students who know LSP, this analogy should click immediately. Before LSP, if you had 10 IDEs and 10 languages, you needed 100 integrations. LSP reduced that to 10+10. MCP does the same thing: if you have 5 AI models and 20 tools, instead of 100 custom integrations, each tool builds one MCP server and each AI model implements one MCP client. The protocol uses JSON-RPC for communication.

MCP Architecture

How it works

  • MCP Client = your AI tool (Claude Code, Cursor, etc.)
  • MCP Server = service wrapper (GitHub, Notion, DB, etc.)
  • Client discovers server capabilities at connection
  • AI decides when to call tools based on your request

Two Transport Types

Remote (HTTP/SSE)
Cloud services, shared servers
e.g., Notion, GitHub, Sentry
Local (stdio)
Local processes, file access
e.g., filesystem, local DB, scripts

MCP is now governed by the Linux Foundation (Agentic AI Foundation) — ensuring it remains an open, vendor-neutral standard.

The architecture is client-server. Claude Code acts as the MCP client. When you connect an MCP server (like GitHub), Claude discovers what tools are available -- for example, create_issue, list_pull_requests, etc. Then when you ask Claude to do something related, it decides which MCP tool to call. There are two transport types: remote (HTTP with Server-Sent Events for streaming) for cloud services, and local (stdio) for processes running on your machine. The fact that it is now under the Linux Foundation signals long-term industry commitment.

Key MCP Servers

A growing ecosystem of ready-to-use integrations

Server What It Does Example Use
GitHub PRs, issues, code review "Review the latest PR"
Sentry Error monitoring "What are today's top errors?"
PostgreSQL Database queries "How many users signed up this week?"
Notion Documentation "Find the API spec document"

This table shows the most commonly used MCP servers. The key insight is that once connected, you talk to these services in natural language. You do not need to know the GitHub API or SQL syntax -- Claude translates your intent into the right tool calls. The PostgreSQL example is particularly powerful: you can ask business questions in plain English and get SQL-backed answers.

Configuring MCP in Claude Code

Add remote servers:


# Add Notion MCP server
claude mcp add --transport http notion \
  https://mcp.notion.com/mcp

# Add GitHub MCP server
claude mcp add --transport http github \
  https://api.githubcopilot.com/mcp/

Manage servers:


claude mcp list         # list all configured servers
/mcp                    # check status inside a session

Three Configuration Scopes

Local
This machine only
--scope local
Project
Shared via .mcp.json
--scope project
User
All your projects
--scope user
Configuration is done with the claude mcp add command. The transport flag tells Claude Code how to connect -- http for cloud services, stdio for local processes. There are three scopes: local (just your machine, not committed), project (stored in .mcp.json which you commit to share with your team), and user (in your home directory, applies to all projects). Use /mcp inside a session to check which servers are connected and their status.

With PostgreSQL, you can ask business questions in plain English and Claude writes the SQL, runs it, and presents the results. This is a huge productivity boost, especially for developers who are not SQL experts or who need quick answers without context-switching.

CLAUDE.md & Prompt Engineering

Teaching Claude how to work on your project

This section covers two related topics: CLAUDE.md (persistent project instructions) and prompt engineering (how to write effective prompts for Claude Code). Both are about communication -- giving Claude the right context to do its best work.

What is CLAUDE.md?

A special markdown file automatically read at every session start

Think of it as persistent project instructions — your team's knowledge base for the AI.

Memory Hierarchy (highest to lowest priority):

Org Managed policy Organization-wide rules
Team ./CLAUDE.md Project memory (committed)
Team .claude/rules/*.md Project rules (topic-specific)
Personal ~/.claude/CLAUDE.md User-wide memory
Local ./CLAUDE.local.md Project local (not committed)
Auto ~/.claude/projects/.../memory/ Automatic session memory
CLAUDE.md is the single most important file for Claude Code productivity. It is auto-loaded every time Claude Code starts in your project directory. The memory hierarchy shows that there are multiple levels: organization policies at the top, then project-level CLAUDE.md shared with your team, then project rules for specific topics, then personal preferences, then local overrides, and finally automatic memory that Claude manages itself. The key distinction is team vs personal: CLAUDE.md is committed to git, CLAUDE.local.md is not.

Writing a Good CLAUDE.md

  • Keep it under 300 lines — concise is better
  • Include commands Claude can't guess: build, test, lint, deploy
  • Document code style deviations from common patterns
  • Use emphasis for critical rules: IMPORTANT:, YOU MUST
  • Bootstrap with /init — Claude scans your project and generates one

# CLAUDE.md

## Build & Test
- Build: `npm run build`
- Test: `npm test -- --watch`
- Lint: `npm run lint` (MUST pass before committing)

## Code Style
- Use TypeScript strict mode everywhere
- Prefer functional components with hooks (no class components)
- IMPORTANT: All API responses must use the Result<T> type

## Architecture
- /src/api/ -- REST endpoints (Express)
- /src/services/ -- Business logic (no direct DB access)
- /src/repositories/ -- Database layer (Prisma)
A good CLAUDE.md is like a good onboarding document for a new team member. It should tell Claude things it cannot figure out by reading the code alone: custom build commands, style rules that deviate from convention, architecture decisions. Use /init to let Claude generate an initial CLAUDE.md by scanning your project -- then refine it. Keep it under 300 lines because it is loaded into every conversation and uses context window space.

Prompt Tip #1: Verify the Work

Always give Claude a way to check its own output

Bad Prompt

implement email validation

No success criteria. Claude guesses what "correct" means.

Good Prompt

Write a validateEmail function.

Test cases:
- user@example.com -> true
- invalid -> false
- test@.com -> false

Run the tests after implementing.

Claude can verify its own work.

Key Principle

If Claude can run tests, check output, or validate results, it will catch its own mistakes. Think of it as giving the AI a feedback loop.

This is the single most impactful prompt engineering tip. When you give Claude test cases and tell it to run them, it creates a feedback loop: write code, run tests, see failures, fix code, run tests again. Without this, Claude writes code and hopes it works. With this, Claude iterates until it verifiably works. This mirrors how good software engineers work -- write tests first, then implement.

Prompt Tip #2: Be Specific, Give Context

  • Use @filename to reference specific files 
    
"refactor @src/auth/login.ts to use async/await"
  • Paste images — screenshots of errors, UI mockups, diagrams
  • Give URLs — Claude can fetch and read web pages 
    
"implement the API from https://docs.example.com/api/v2"
  • For complex features, let Claude interview you
Vague

"add authentication"

Specific

"add JWT authentication to @src/api/routes.ts using the existing User model in @src/models/user.ts. Use bcrypt for password hashing. Include refresh token rotation."

The more context you give Claude, the better the output. The @ syntax is a shortcut to reference files -- Claude will read them before starting. You can also paste screenshots of errors or UI mockups directly into the conversation. URLs work too -- Claude will fetch the page and use the content. The key mental shift: treat Claude like a skilled developer who just joined your team today. They can write great code, but they need to know your specific context.

Context Management

Claude has a 1M token context window — but managing it well matters

Essential Commands

  • /compact — Summarize conversation, free up space
  • /clear — Full reset between tasks
  • Auto-compaction at ~83% capacity

Advanced Techniques

  • Subagents — Separate context for investigation tasks
  • Extended thinking — Use "ultrathink" keyword for complex reasoning
  • --continue — Resume previous conversations
When to /compact vs /clear

/compact: Same task, running low on context — keeps a summary of what happened.
/clear: Switching to a completely different task — starts fresh.

Think of context like RAM: /compact = writing to swap, /clear = reboot

Context management is a practical skill that separates productive Claude Code users from frustrated ones. The 1M token window sounds huge, but complex tasks with many file reads fill it up fast. Use /compact when you are mid-task but running low -- it summarizes the conversation so far. Use /clear when switching to a completely different task. Subagents are like spawning a child process with its own memory -- useful for "go investigate this and report back" tasks. The "ultrathink" keyword triggers extended thinking mode, where Claude spends more time reasoning before responding -- useful for complex architecture decisions.

The Interview Pattern

Let Claude ask you the right questions


I want to build [description of your feature].

Interview me in detail about this feature.
Ask about:
- Technical implementation details
- UI/UX requirements
- Edge cases and error handling
- Performance considerations

Keep interviewing until everything is covered,
then write a complete spec to SPEC.md.

Why this works

  • Claude asks questions you would not think of
  • Surfaces edge cases before implementation
  • Produces a complete spec that guides all subsequent work
  • Shifts Claude from "guess what they want" to "know what they want"

This is especially powerful for features where you have a rough idea but have not thought through all the details.

The interview pattern is one of the most underused techniques. Instead of trying to write a perfect prompt that covers everything, you let Claude interview you. Claude will ask about things you have not considered -- error handling, edge cases, accessibility, performance. After the interview, it writes a complete spec that both of you agree on. This is like having a senior engineer do a requirements gathering session with you before coding starts.

Skills, Tools & Hooks

Extending Claude Code's capabilities

This section covers three ways to extend Claude Code beyond its defaults: Skills (reusable prompt templates), Tool permissions (security model), and Hooks (lifecycle automation). These are more advanced features but very powerful once understood.

Skills: Reusable Prompt Templates

Markdown files that extend Claude with specialized behavior

  • Stored as SKILL.md files
  • Invoked via /skill-name (slash commands)
  • Can define allowed tools, context files
  • Dynamic context with !`command` syntax

---
name: review-pr
description: Review a pull request
allowed-tools: [Bash, Read, Grep]
---

# PR Review Skill

## Context
Current branch: !`git branch --show-current`
Changed files: !`git diff --name-only main`

## Instructions
1. Read each changed file
2. Check for:
   - Security issues
   - Performance problems
   - Missing tests
3. Write review as a comment
Dynamic Context Injection

The !`command` syntax runs a shell command and injects the output into the skill at load time. This means the skill always has fresh context about the current state of your project.

Skills are like reusable recipes for Claude. Instead of typing the same complex prompt every time you review a PR, you create a skill file that encapsulates the process. The frontmatter defines metadata like which tools the skill can use. The dynamic context syntax with backticks is powerful -- it runs a command and injects the output when the skill loads, so Claude always has current information. Skills were merged with slash commands in version 2.1.3, so you invoke them the same way.

Tool Use & Permissions

Claude Code's built-in tools and security model

Built-in Tools

ReadRead files
GlobFind files by pattern
GrepSearch file contents
EditModify files
WriteCreate new files
BashRun shell commands
WebFetchRead web pages

Permission Modes

default
Ask before writes/commands
acceptEdits
Auto-approve file edits, ask for Bash
plan
Read-only, no modifications
bypassPermissions
Auto-approve all (containers only!)

Security note: Shell operators (&&, |, ;) do not bypass permission rules — each command in a chain is checked individually.

Claude Code has a set of built-in tools that it uses to interact with your project. Each tool has a specific purpose -- Read for reading files, Glob for finding files, Grep for searching content, etc. The permission model controls what Claude can do without asking. In default mode, it asks before every write operation and shell command. acceptEdits lets it freely edit files but still asks for shell commands. plan mode is read-only -- great for understanding code without risk of changes. bypassPermissions is only for containers where there is nothing sensitive to protect.

Hooks: Lifecycle Automation

User-defined shell commands triggered at key events

PreToolUse Before a tool runs — Block dangerous commands
PostToolUse After a tool runs — Auto-format, lint, validate
SessionStart Session begins — Inject context, check env
Notification Claude sends notification — Desktop alerts, sounds
Stop Claude finishes a turn — Verify completion, run checks 

// .claude/settings.json -- auto-format after every file edit
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "command": "npx prettier --write $CLAUDE_FILE_PATH"
    }]
  }
}

Exit code 0 = proceed, exit code 2 = block the action

Hooks are a powerful automation mechanism. They let you run custom shell commands at specific points in Claude Code's lifecycle. The most common use case is PostToolUse with a formatter: every time Claude writes or edits a file, the hook automatically runs Prettier or your formatter. PreToolUse can block dangerous commands before they execute -- for example, preventing rm -rf or git push --force. The exit code convention is simple: 0 means proceed normally, 2 means block the action. This is inspired by git hooks.

Agent Teams is an experimental feature where you can run multiple Claude Code instances that work on different parts of your project simultaneously, like having multiple developers.

Best Practices Summary

1 Start with /init to bootstrap your CLAUDE.md
2 Use plan mode (Shift+Tab) before coding complex features
3 Include verification steps (test cases, checks) in every prompt
4 Connect MCP servers for GitHub, Sentry, databases, and more
5 Create skills for repeated workflows (PR review, deploy, etc.)
6 Use /compact and /clear to manage context wisely
These six practices will take students from beginner to productive Claude Code users. The most important ones are: always start with CLAUDE.md (it dramatically improves output quality), always include verification steps in prompts, and use plan mode for anything non-trivial. MCP and skills are force multipliers once the basics are solid.
