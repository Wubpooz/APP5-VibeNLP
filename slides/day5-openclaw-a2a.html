From Chatbots to Agents

The evolution of AI systems (2022–2026)

2022 ChatGPT — conversational chat, responds to input
2023 GPT-4 + Plugins — LLMs gain access to external tools
2024 Claude Code + MCP — agentic coding, tool protocols emerge
2025 OpenClaw + Moltbook — autonomous agents on messaging platforms
2026 A2A protocols — multi-agent systems, agents talking to agents
The Key Shift

Chatbots wait for your input. Agents act proactively — they plan, execute, use tools, and collaborate without constant human supervision.

Frame the evolution for students who are new to AI. 2022: ChatGPT was a chatbot - you ask, it answers. 2023: GPT-4 plugins let LLMs browse the web, run code. 2024: Claude Code could autonomously write and execute code. MCP standardized tool access. 2025: OpenClaw turned LLMs into always-on personal assistants on messaging platforms. 2026: A2A lets these agents delegate work to OTHER agents. The trajectory is clear: from reactive chatbots to proactive, collaborative agents.

How OpenClaw connects everything together

Key: the gateway runs locally. Your data stays on your machine. Only LLM API calls leave your network.

Walk through the architecture layer by layer. You chat on WhatsApp (or another platform). The message goes to the OpenClaw gateway running on your own machine. The gateway sends the message to an LLM for reasoning, then executes actions locally. The key insight: the LLM is the "brain" (in the cloud), but the "hands" (filesystem, shell, tools) are all local. This is the "local-first" philosophy. Model-agnostic means you just swap API keys - bring your own brain.

All files are plain Markdown — no YAML, no JSON, no database

This is the core of OpenClaw's design. Every aspect of the agent is defined by a Markdown file. Walk through each one: - AGENTS.md: the operating manual (rules, constraints) - SOUL.md: personality, tone, what the agent should/shouldn't do - TOOLS.md: available tools and how to use them - BOOTSTRAP.md: runs only on first launch (setup wizard) - IDENTITY.md: the agent's name and self-description - USER.md: who the user is, their preferences - HEARTBEAT.md: periodic tasks (every ~30 min) - memory/*.md: daily conversation logs in Markdown

Design Philosophy: Files as Config

Plain Markdown is the single source of truth

Traditional Agent Frameworks OpenClaw
Configuration YAML / JSON / TOML configs Markdown files (.md)
Memory Vector database (Pinecone, etc.) Dated .md log files
State Opaque binary state / SQLite Human-readable text
Debugging Custom logging, dashboards grep / cat / less
Portability Export/import, migrations Copy the folder
Version control Often difficult git add && git commit
Why this matters

If you can read a text file, you can understand, debug, and modify your agent. No proprietary format, no vendor lock-in, no hidden state.

This is OpenClaw's most radical design decision. Every other agent framework uses structured config (YAML, JSON) and databases for memory. OpenClaw uses just plain text. Why? Because Markdown is what LLMs understand best. It's also what humans read most easily. And it works perfectly with git. Analogy for students: think of it like Unix philosophy - text is the universal interface. Pipes work because everything is text.

Context Loading: Main vs Sub-Agent

Not all agents get the same context

File Purpose Main Agent Sub-Agent
AGENTS.md Operating instructions ✓ Yes ✓ Yes
SOUL.md Persona & tone ✓ Yes ✗ No
TOOLS.md Tool notes ✓ Yes ✓ Yes
BOOTSTRAP.md First-run ritual ★ First run ✗ No
IDENTITY.md Agent name & vibe ✓ Yes ✗ No
USER.md User profile ✓ Yes ✗ No
HEARTBEAT.md Periodic checklist ↻ Every ~30m ✗ No
memory/*.md Session memory ✓ Yes ✗ No
Sub-agents don't need a soul

Sub-agents are task executors, not conversationalists. Lean context = better performance. Extra info comes via the spawn prompt.

This is a key architectural decision. The main agent is the "face" of the system: it has personality, knows the user, has memory. Sub-agents are specialists spawned for specific tasks - they only need the rules (AGENTS.md) and the tools (TOOLS.md). Analogy: think of the main agent as the CEO and sub-agents as contractors. The CEO knows the company culture and all employees. Contractors get a task brief and the tools to do the job. Lean context means more room in the context window for the actual task.

A2A Protocol

How Agents Talk to Each Other

Transition to the A2A section. We've seen how a single agent works. Now: how do MULTIPLE agents communicate and collaborate?

The Protocol Problem

How do independent AI agents communicate?

Agent A

Own memory, own tools, own context

?
Agent B

Own memory, own tools, own context

  • They don't share memory or context — they're "opaque" to each other
  • Discovery: How does Agent A find Agent B? What can B do?
  • Communication: What message format? What transport?
  • Task management: How to delegate work and track progress?
  • Security: How to authenticate and authorize?

Analogy: imagine two people who speak different languages, don't know each other, and need to collaborate on a project. You need a shared protocol.

Before A2A, there was no standard way for agents to talk to each other. Each agent framework had its own approach, if any. Imagine: your OpenClaw agent needs a specialized agent to do legal research. How does it find it? How does it send a request? How does it get the result? This is exactly the problem that networking protocols solved for computers (TCP/IP, HTTP). Now we need the same for AI agents.

A2A: The Protocol

Open protocol by Google (April 2025), now under the Linux Foundation

100+

Supporting companies

1. Discovery

Agent Cards
Published at
/.well-known/agent.json

2. Communication

HTTP + JSON-RPC 2.0
Streaming via
Server-Sent Events (SSE)

3. Task Management

Task lifecycle
submitted → working
→ completed / failed

Think of A2A like HTTP for agents: a standard protocol that lets any agent talk to any other agent, regardless of who built them.

A2A has three layers, similar to how web protocols work: 1. Discovery - like DNS. How do you find an agent? Agent Cards at a well-known URL. 2. Communication - like HTTP. Standard message format (JSON-RPC) over HTTP. 3. Task management - like job queues. Submit work, track status, get results. The three pillars map to familiar concepts for software engineers: - Agent Cards = API documentation / service discovery - JSON-RPC = REST API / gRPC - Task lifecycle = job queue (Celery, RabbitMQ)

Agent Cards

A business card for AI agents

Published at /.well-known/agent.json

{
  "name": "Legal Research Agent",
  "description": "Searches case law
    and summarizes findings",
  "url": "https://legal.example.com/a2a",
  "version": "1.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "skills": [
    {
      "id": "case-search",
      "name": "Case Law Search",
      "description": "Search by topic,
        jurisdiction, or citation"
    }
  ],
  "authentication": {
    "schemes": ["Bearer"]
  }
}

What it contains:

  • Name & description
  • Endpoint URL (where to send requests)
  • Skills (what it can do)
  • Capabilities (streaming, push, etc.)
  • Authentication (how to prove identity)
The Business Card Analogy

When you meet someone, they hand you a card: name, title, phone, email. An Agent Card does the same — but for AI agents.

Agent Cards are the first step in agent-to-agent communication. Just like /.well-known/security.txt tells you how to report security issues, /.well-known/agent.json tells other agents what this agent can do. For software engineers: think of it like an OpenAPI/Swagger spec, but for AI agents instead of REST APIs. The business card analogy makes it intuitive: at a networking event, you exchange cards to learn who does what and how to contact them.

A2A Task Flow

How a client agent delegates work to a remote agent

Client Agent → GET /.well-known/agent.json  (fetch Agent Card) Remote Agent
← Agent Card (name, skills, auth)  (JSON response)
Client Agent → tasks/send  (JSON-RPC: create task with instructions) Remote Agent
← Status: working...  (streamed via SSE)
← Status: completed + artifacts  (final result)

Task lifecycle: submitted → working → completed | failed | canceled

Walk through the flow step by step: 1. Client agent discovers the remote agent by fetching its Agent Card 2. Reads the card to understand what the agent can do 3. Creates a task with instructions (JSON-RPC call) 4. Remote agent starts working, streams progress via SSE 5. Returns the final result with artifacts (documents, data, etc.) For software engineers: this is exactly like calling a REST API, but with built-in streaming (SSE) and task lifecycle management. The remote agent can take seconds or hours - the protocol handles both.

This is THE key diagram of the entire presentation. MCP is VERTICAL: it connects an agent to tools (like plugging devices into a USB port). A2A is HORIZONTAL: it connects agents to each other (like a telephone network). MCP = "what can I do?" (tools, APIs, databases) A2A = "who can help me?" (other specialized agents) For students: USB port vs telephone network is the key analogy. Both are now governed by the Linux Foundation, making them neutral standards. In practice, a sophisticated agent would use BOTH: MCP for its own tools, and A2A to delegate tasks to other agents.

For software engineers: these map to familiar architecture patterns - monolith, microservices with gateway, ETL pipeline, and P2P network.

Security Lessons

What OpenClaw teaches us about agent security

1. Open-source velocity vs security
180K stars in 3 weeks, but security auditing can't keep pace with adoption. Popularity is not a proxy for safety.
2. Agent permissions = new attack surface
Agents that can execute code, access files, and browse the web create attack surfaces far beyond traditional software.
3. Supply chain attacks on skill marketplaces
386 malicious skills in ClawHub. Just like npm/PyPI malware, AI skill stores face the same supply chain risks.
4. Defensive measures
Always update. Validate skill origins. Run agents in containers. Treat localhost as untrusted.
Key Principle

localhost is not a security boundary. An agent running on your machine with shell access is a powerful weapon — for you or an attacker.

These four lessons are essential for any software engineer working with AI agents: 1. Just because a project is popular doesn't mean it's secure. Open-source projects can grow faster than security reviews can happen. 2. Traditional software might read/write files. Agents can execute arbitrary code, browse the web, send messages. The attack surface is enormous. 3. Supply chain attacks are already a known problem (npm, PyPI). Now the same thing happens with AI skill marketplaces. 4. Practical advice: update regularly, run in containers (Docker), validate where skills come from, and never assume localhost is safe. The localhost point is especially important: many developers think "it's on my machine, so it's safe." WebSocket hijacking proves this wrong.

Agent Evolution

What If Agents Could Learn From Each Other?

Transition to evolution section. We've covered connection (MCP), execution (Skills), and communication (A2A). Now: the final piece - Evolution.

The Repetition Problem

Every agent learns from scratch

Day 1: Your agent hits a timeout error. Figures out exponential backoff.
Day 2: Another agent hits the same error. Figures it out again.
Day 3: 10,000 agents hit the same error. Each solves it independently.

$10,000

Token cost of redundant trial-and-error

$200

If solutions were shared

Analogy: imagine if every programmer had to rediscover sorting algorithms from scratch.

This is the core insight: agents today are like isolated individuals, each solving the same problems over and over. We've built amazing protocols for agents to communicate (A2A), but they can't share EXPERIENCE. The economic argument is compelling: if 10,000 agents each spend $1 in tokens solving the same error, that's $10,000 wasted. But if one agent solves it and shares the solution, the total cost drops to ~$200.

The Missing Layer

We have connection, execution, communication... what about evolution?

MCP Connection Tells agent what tools exist
Skills Execution Tells agent how to do things
A2A Communication Lets agents talk to each other
GEP Evolution Lets agents share experience
GEP: Genome Evolution Protocol

A protocol that lets agents package their solutions, publish them to a network, and inherit solutions from other agents.

GEP completes the picture. We've built the infrastructure for agents to exist (MCP), to know how to do things (Skills), and to talk to each other (A2A). But we haven't built infrastructure for them to EVOLVE - to accumulate collective experience over time. GEP fills this gap.

Gene

A strategy template
"When you see TimeoutError, use exponential backoff"

Capsule

A validated implementation
Actual code + success rate + environment info

The terminology is deliberately biological: - Gene = a reusable strategy template (like a "recipe" for solving a type of problem) - Capsule = a validated implementation of a gene (like a "tested dish" with reviews) The flow: 1. Agent A encounters an error 2. It experiments and finds a solution 3. The solution is packaged as a Capsule (with metadata: success rate, environment) 4. Capsule is published to the network 5. Agent B encounters similar error, searches network, finds and inherits the solution Like biological evolution: good solutions spread, bad ones die out.

The Evolution Mindset

Building agents that learn collectively

  • Before solving a problem: Search the network first
    Maybe another agent already solved it
  • After solving a problem: Package and share
    Your solution could help thousands of other agents
  • Let bad solutions die: The network ranks by quality
    Good solutions rise, bad ones get buried
Key Insight

One agent learns, a million agents inherit.
This is the promise of agent evolution — collective intelligence that compounds over time.

The mindset shift is important: 1. Don't reinvent the wheel - search first 2. Don't hoard solutions - share them 3. Trust the network to surface the best solutions This is how biological evolution works: successful adaptations spread, unsuccessful ones disappear. We're applying the same principle to AI agents. The vision: over time, the collective intelligence of all agents grows, even though each individual agent starts from scratch.
