What Is a Prompt?

A prompt is the input text you provide to an LLM to elicit a desired output.

Prompt (your input)
→
LLM
→
Completion (output)

A prompt can include:

  • Instructions — What you want the model to do
  • Context — Background information, documents, data
  • Examples — Input/output pairs showing the pattern
  • Constraints — Format, length, style requirements
At its core, a prompt is just text input. But the quality of that text determines the quality of the output. Think of it as an API call where the interface is natural language. A good prompt typically combines several of these elements: instructions, context, examples, and constraints. We will see concrete examples of each throughout this session.

Part 2

Zero-Shot and Few-Shot Prompting

Now let's get into the two foundational prompting strategies. Everything else builds on these.

Zero-Shot Prompting

Just describe the task. No examples needed.

User:
Classify the sentiment of this review as positive, negative, or neutral:

"The battery life is incredible but the screen is too dim outdoors."
Assistant:
Neutral (mixed). The review expresses a positive opinion about battery life but a negative opinion about screen brightness.

Works well for tasks the model has seen during pre-training (classification, summarization, translation).

Zero-shot is the simplest approach: just tell the model what to do. Modern LLMs are remarkably good at this for common tasks because they've seen millions of similar patterns during pre-training. GPT-4, Claude, Gemini all handle basic classification, summarization, and translation in zero-shot mode quite well. But for unusual or complex tasks, zero-shot often falls short. That's where few-shot comes in.

Few-Shot Prompting

Provide examples to teach the pattern.

User:
Extract the product and issue from support tickets.

Ticket: "My AirPods Pro keep disconnecting from my iPhone."
Output: {"product": "AirPods Pro", "issue": "disconnecting"}

Ticket: "The MacBook keyboard stopped working after the update."
Output: {"product": "MacBook", "issue": "keyboard not working"}

Ticket: "iPad screen flickers when brightness is above 50%."
Output:
Assistant:
{"product": "iPad", "issue": "screen flickering"}

This is In-Context Learning (ICL) — the model learns the task from examples in the prompt, with no weight updates. (Brown et al., 2020)

Few-shot prompting is one of the most powerful ideas in modern NLP. You give the model a few input-output examples, and it figures out the pattern. No training, no fine-tuning. This is called In-Context Learning, introduced in the GPT-3 paper by Brown et al. The model's weights don't change — it's doing pattern matching in the context window. Notice how the output format is exactly consistent with the examples. That's the power of ICL.

In-Context Learning: Key Questions

How many examples?

  • 1-3 examples: Usually enough for simple tasks
  • 4-8 examples: Complex extraction or classification
  • Diminishing returns beyond 8-10
  • More examples = more tokens = higher cost

Where to place them?

  • Before the actual query (standard)
  • Most models attend more to recent tokens
  • Put the most similar example closest to the query
  • Consistent formatting across all examples
Two practical questions every engineer asks: how many examples do I need, and where do I put them? For most tasks, 2-5 examples is the sweet spot. Research shows diminishing returns beyond that, and you're burning tokens (and money) for minimal gains. Placement matters: recency bias means the model pays more attention to examples near the query. So put your best, most representative example last, right before the actual input. Also, consistency in formatting is crucial. If your examples have slightly different formats, the model will be confused about which pattern to follow.

Example Selection Strategies

Strategy How It Works When to Use
Random Pick examples randomly from a pool Baseline; quick prototyping
Similarity-based Embed query, find nearest examples Diverse input space; retrieval-augmented
Diverse Cover different categories / edge cases Classification with many classes
Difficulty-based Include hard examples the model gets wrong When model struggles with edge cases

Liu et al. (2022) showed that example selection can matter more than example count.

Not all examples are created equal. Liu et al. demonstrated that selecting the right examples can improve few-shot performance by 10-20% compared to random selection. In practice, similarity-based selection (using embeddings to find the most relevant examples for each query) works best for production systems. This is basically RAG for few-shot examples. For classification, make sure to cover all classes. For extraction, show diverse input patterns. The key insight: treat your example bank like training data — curate it carefully.

Part 3

Prompt Engineering Best Practices

Now let's talk about the craft of writing good prompts. These are patterns that consistently improve results across all LLMs.

Rule 1: Be Specific and Clear

Summarize this article.

Vague: what length? what focus? what style?

Summarize this article in 3 bullet points, focusing on the technical contributions. Each bullet should be one sentence. Write for a software engineer audience.

Specificity checklist:

  • Output format — length, structure, style
  • Audience — who is the output for?
  • Scope — what to include / exclude
  • Tone — formal, casual, technical
The number one mistake in prompting is being too vague. The model fills in ambiguity with its own assumptions, and those assumptions may not match yours. Think of it like giving instructions to a very capable but very literal intern. They'll do exactly what you ask — so you need to ask precisely. The good prompt here specifies format (bullet points), count (3), focus (technical contributions), length (one sentence each), and audience (software engineers). That's 5 constraints in 2 sentences.

Rule 2: Provide Context & Constraints

User:
You are a senior code reviewer at a fintech company.
Our codebase uses Python 3.11, FastAPI, and SQLAlchemy.
We follow PEP 8 and require type hints on all public functions.

Review the following function. Focus on:
1. Security vulnerabilities (especially SQL injection)
2. Error handling
3. Type safety

Format your response as:
- CRITICAL: [issues that must be fixed]
- WARNING: [issues that should be fixed]
- SUGGESTION: [nice-to-have improvements]

```python
def get_user(db, user_id):
    query = f"SELECT * FROM users WHERE id = {user_id}"
    return db.execute(query).first()
```

Role + Tech context + Focus areas + Output format = Excellent prompt

This is a production-quality prompt. Notice the layered structure: First, we set a ROLE (senior code reviewer at fintech) — this activates relevant knowledge. Then CONTEXT (Python 3.11, FastAPI, SQLAlchemy, PEP 8) — this narrows the solution space. Then FOCUS AREAS (security, error handling, types) — this prioritizes what to look for. Finally, OUTPUT FORMAT (CRITICAL/WARNING/SUGGESTION) — this makes output actionable. The model will flag the SQL injection vulnerability as CRITICAL, the missing type hints as WARNING, and maybe suggest using ORM queries as a SUGGESTION. That's exactly what you want from a code review.

Rule 3: Use Structured Output Formats

JSON output

Extract entities from this text. Return a JSON object with this schema:

{
  "people": [{"name": str, "role": str}],
  "organizations": [str],
  "dates": [{"date": str, "event": str}]
}

XML tags for separation

Analyze this code and respond using these tags:

<analysis>Your analysis</analysis>
<fixed_code>Corrected code</fixed_code>
<explanation>What you changed and why</explanation>

Why structured output matters:

  • Easier to parse programmatically
  • Reduces ambiguity in model responses
  • Enables downstream processing (API pipelines)
  • Many APIs now support forced JSON mode
When building production systems, you need machine-readable output. JSON is the most common choice because every language can parse it. XML tags are great when you need to separate different sections of a response — Anthropic's Claude documentation actually recommends this approach. Most modern LLM APIs (OpenAI, Anthropic, Google) now support JSON mode or structured outputs, which guarantees valid JSON. Always use that in production. Pro tip: provide the exact schema you expect. Don't just say "return JSON" — show the model the keys, types, and nesting you want.

System Prompts vs User Prompts

System (persistent instructions):
You are a helpful assistant that specializes in Python code review.
Always respond in this format: ISSUE → FIX → EXPLANATION.
Never suggest changes that break backward compatibility.
User (per-request input):
Review this function:
```python
def process(data): return [x*2 for x in data if x > 0]
```
System PromptUser Prompt
ScopeApplies to all messagesSingle request
ContentRole, rules, format, constraintsSpecific task + input data
AnalogyJob descriptionToday's task
Modern LLM APIs separate system prompts from user prompts. Understanding the difference is critical. The system prompt sets the stage: who is the model, what rules does it follow, what format to use. It persists across the conversation. Think of it as the model's job description. The user prompt is the specific request. It changes with each interaction. Best practice: put all persistent instructions, format requirements, and constraints in the system prompt. Put variable content (the actual data to process) in the user prompt. This separation makes your prompts modular and reusable.

Temperature & Sampling Parameters

T=0
Deterministic

Code, math, extraction
Always pick top token

T=0.7
Balanced

General conversation
Creative but coherent

T=1.0+
Creative

Brainstorming, writing
High variety, more risk

* Range varies by model: Claude 0–1, OpenAI 0–2

Other parameters to know:

  • top_p (nucleus sampling) — threshold for cumulative probability
  • max_tokens — hard cap on output length
  • stop sequences — custom tokens that end generation
Temperature controls randomness. At T=0, the model always picks the highest-probability token. This is what you want for factual tasks: code generation, data extraction, math. At T=0.7, there's some randomness — good for general conversation where you want natural variation. Above 1.0, the distribution flattens and you get more creative but less reliable output. Rule of thumb: if there's one correct answer, use T=0. If you want variety, use T=0.7-1.0. top_p is an alternative to temperature. Using both at non-default values is not recommended. max_tokens prevents runaway generation. stop sequences are useful for structured output.

Part 4

Chain-of-Thought Prompting

Wei et al., "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models" (2022)

Now we get to one of the most important breakthroughs in prompting research. Chain-of-Thought fundamentally changed what we thought LLMs could do.

Why Do LLMs Give Wrong Answers?

🎯 Greedy Decoding (default)

At every step, pick the single most probable next token

Q: 17 × 24 = ?
→ Model sees "408" as most likely answer token
→ Outputs: 408 ✗ (correct: 408... wait, is it? 😅)

Like answering before thinking — System 1

💡 The Key Insight (Denny Zhou, 2024)

Reasoning-based outputs don't appear at the top of the distribution

The model already knows how to reason
(trained on millions of human step-by-step solutions)

But greedy decoding suppresses those paths

The capability is there — we just need to unlock it

The question: How do we get the model to "slow down and think" instead of blurting out the first thing that comes to mind?
This is the motivational heart of Chain-of-Thought — the "why" that makes everything else make sense. By default, LLMs use greedy decoding: at each step, they pick the single most probable next token. This is like answering a question before you've had time to think about it — System 1 thinking, in Kahneman's terms. Fast, automatic, and often wrong for complex problems. Denny Zhou (DeepMind Chief Scientist, creator of CoT) made a crucial observation in his 2024 paper "Chain-of-Thought Reasoning Without Prompting" (arXiv:2402.10200): reasoning-based outputs don't appear at the TOP of the probability distribution. So greedy decoding never generates them. BUT — and this is the key — the model already has the reasoning capacity! It was trained on enormous amounts of human writing that includes step-by-step reasoning, mathematical proofs, code comments, etc. That knowledge is encoded in the weights. Greedy decoding just doesn't give it a chance to show. So the problem isn't that the model can't reason. The problem is that we're not letting it. CoT prompting is the first answer to this problem. Reference: Wang & Zhou (2024), "Chain-of-Thought Reasoning Without Prompting", NeurIPS 2024 arXiv: https://arxiv.org/abs/2402.10200

The Breakthrough

Standard prompting

User:
Roger has 5 tennis balls. He buys 2 more cans of tennis balls. Each can has 3 balls. How many tennis balls does he have now?
Assistant:
The answer is 11. ✗

Chain-of-Thought

User:
Roger has 5 tennis balls. He buys 2 more cans of tennis balls. Each can has 3 balls. How many tennis balls does he have now?
Let's think step by step.
Assistant:
Roger starts with 5 balls.
He buys 2 cans × 3 balls = 6 balls.
5 + 6 = 11. ✓

Adding "Let's think step by step" improved accuracy on GSM8K math benchmark from 17.7% to 78.7%.

Kojima et al. (2022): "Large Language Models are Zero-Shot Reasoners"

This is the famous example from the Chain-of-Thought paper. In this simple case both get 11, but the reasoning trace matters enormously for harder problems. The key insight: when you ask the model to show its work, it actually reasons better. It's not just producing prettier output — it makes fewer errors on multi-step problems. The "Let's think step by step" phrase was shown by Kojima et al. to be a remarkably effective zero-shot CoT trigger. On GSM8K (grade school math), accuracy went from 17.7% to 78.7%. That's a 4x improvement from six words!

Zero-Shot CoT vs Few-Shot CoT

Zero-Shot CoT

[Problem statement]

Let's think step by step.
  • Simple to use
  • No examples needed
  • Works well for common reasoning

Few-Shot CoT

Q: [Example problem 1]
A: [Step-by-step solution 1]

Q: [Example problem 2]
A: [Step-by-step solution 2]

Q: [Actual problem]
A:
  • More reliable for complex tasks
  • Controls reasoning style
  • Best for domain-specific reasoning
There are two flavors of Chain-of-Thought. Zero-shot CoT is the easy one: just append "Let's think step by step" to your prompt. Few-shot CoT is more powerful: you provide examples that include the reasoning steps. The model then mimics your reasoning style. This is crucial for domain-specific tasks where the reasoning pattern is not obvious from the question alone. For example, if you're doing legal analysis, you might show examples of how to reason through statute interpretation. The model will follow that specific reasoning pattern. In practice, zero-shot CoT is your default. Switch to few-shot CoT when you need more control over the reasoning process.

CoT in Practice: Coding Example

User:
What is the time complexity of this function? Think step by step.

```python
def find_pairs(arr, target):
    seen = set()
    pairs = []
    for num in arr:
        complement = target - num
        if complement in seen:
            pairs.append((complement, num))
        seen.add(num)
    return pairs
```
Assistant:
Step 1: Identify the loop → one `for` loop over `arr` of length n.
Step 2: Inside the loop: `complement = target - num` is O(1).
Step 3: `complement in seen` → set lookup is O(1) average.
Step 4: `pairs.append(...)` is amortized O(1). `seen.add(num)` is O(1).
Step 5: All operations inside the loop are O(1), loop runs n times.
Answer: O(n) time complexity, O(n) space for the set.
Here's a practical software engineering example. The model breaks down the complexity analysis into discrete steps, examining each operation inside the loop. Without CoT, models sometimes just guess "O(n^2)" because they see a list-searching pattern. With CoT, the model correctly identifies that set operations are O(1) on average, leading to the correct O(n) analysis. This is the kind of prompt you'd use in code review, interview prep, or teaching.

Why Does CoT Work? — Two Levels

Surface level (the engineering view)

  • Decomposition — breaks multi-step problems into single-step ones
  • Working memory — context window acts as scratchpad; intermediate results persist
  • Error localization — you can see where reasoning goes wrong
  • Autoregressive priming — each generated token shifts probability of the next toward coherent reasoning

Deeper truth (Denny Zhou's insight)

Pre-trained LLMs already possess inherent reasoning capacity

They were trained on human step-by-step problem-solving

CoT prompting shifts probability mass toward reasoning paths that greedy decoding would suppress

More tokens = more test-time compute = better access to correct answers

Wang & Zhou, "CoT Reasoning Without Prompting", NeurIPS 2024

CoT helps most with multi-step reasoning. For simple factual recall, it wastes tokens.

Let's look at CoT from two angles — the engineering explanation you'll use in practice, and the deeper theoretical insight from Denny Zhou. The surface-level explanation: CoT works because it decomposes hard problems, uses the context window as working memory, lets you localize errors, and primes the autoregressive process toward coherent reasoning chains. But the DEEPER insight — which Denny Zhou proved experimentally in his 2024 NeurIPS paper — is more profound: pre-trained LLMs already have the reasoning capacity encoded in their weights. They were trained on massive amounts of human text that includes step-by-step problem-solving, mathematical derivations, code with comments, etc. The problem is that greedy decoding suppresses this capacity. Reasoning-based answers don't appear at the top of the probability distribution. So the model "knows" how to reason, but its default decoding prevents it from showing that. CoT prompting works by shifting the probability distribution. The phrase "Let's think step by step" makes the model generate tokens like "Step 1:" which then prime the rest of the reasoning chain. You're essentially unlocking a capability that was always there. This insight led directly to "test-time compute" — the idea that you can get better answers by generating more tokens (thinking longer), rather than always trying to answer in one shot. This is the seed from which o1, DeepSeek-R1, and all reasoning models grew.

One Insight → A Whole Revolution

Denny Zhou founded Google Brain's reasoning team in 2021. Here's what grew from it:

2022
CoT Prompting — "Let's think step by step" (6 words, 4× accuracy jump) Wei et al. · Kojima et al.
2022
Self-Consistency — Sample many reasoning paths, vote on majority answer Wang et al.
2023
Process Reward Models — Train verifiers to score each reasoning step, not just the final answer Lightman et al. (OpenAI)
2024
RLVR + o1 — Reinforce reasoning via verifiable rewards; bake "slow thinking" into the model OpenAI o1
2025
DeepSeek-R1 / Open Reasoning — RLVR at scale, open weights, reasoning democratized DeepSeek et al.
The unifying thread: more thinking = better answers. From a 6-word prompt trick to trillion-dollar AI infrastructure — same core idea.
Here's the beautiful narrative arc that Denny Zhou's Stanford lecture traces. It all started with a simple question: why does "Let's think step by step" work so well? That curiosity, starting around 2021-2022, spawned an entire research agenda. CoT prompting showed that just asking for steps dramatically improved accuracy. Then Self-Consistency showed you could do even better by sampling multiple reasoning paths and taking the majority vote — more compute at inference time, better answers. Process Reward Models took it further: instead of just checking the final answer, train a separate model to evaluate each STEP in the reasoning chain. This gives much better signal for training. RLVR (Reinforcement Learning with Verifiable Rewards) — which we cover in Day 5 — combined these ideas: use verifiable rewards (math answers are right or wrong) to reinforce the reasoning process itself. OpenAI's o1 used this to create a model that "thinks" for seconds or minutes before answering. DeepSeek-R1 in 2025 showed this could be done open-source, democratizing reasoning models. The through-line is always the same: give the model more token budget to think, and it performs dramatically better. The "thinking" is test-time compute. This isn't magic. It's a fundamental insight: the model already has the knowledge, we're just giving it the space to use it. Source: Denny Zhou, "LLM Reasoning", Stanford CS 25, 2025 https://dennyzhou.github.io/LLM-Reasoning-Stanford-CS-25.pdf

Part 5

Advanced Prompting Techniques

Let's look at techniques that build on CoT to push the boundaries even further. These are research-driven approaches that are increasingly used in production.

Self-Consistency

Wang et al., "Self-Consistency Improves Chain of Thought Reasoning" (2022)

Same Prompt
→
Sample N paths
(T > 0)
→
Path 1: Answer A
Path 2: Answer A
Path 3: Answer B
→
Majority vote
Answer A

Key idea:

  • Run the same CoT prompt multiple times with T > 0
  • Different reasoning paths may reach different answers
  • Take the majority vote — the most common answer wins
  • Improves accuracy by 5-15% over single-path CoT

Trade-off: N× more API calls = N× cost. Typical N = 5–10.

Self-Consistency is beautifully simple. Instead of generating one chain of thought, generate several. Use temperature > 0 so each path is different. Then take the majority vote on the final answer. The intuition: if multiple independent reasoning paths agree on an answer, it's likely correct. It's like asking 5 experts and going with the consensus. Wang et al. showed 5-15% improvement over single-path CoT on math and reasoning benchmarks. The downside is cost: 5 samples means 5x the API cost. So use it for high-stakes decisions where accuracy matters more than latency and cost.

Tree of Thoughts (ToT)

Yao et al., "Tree of Thoughts: Deliberate Problem Solving with LLMs" (2023)

Chain-of-Thought (linear)

Thought 1
↓
Thought 2
↓
Thought 3
↓
Answer

Tree of Thoughts (branching)

Start
↙    ↓    ↘
A (prune) B (explore) C (prune)
   ↙  ↘
B1 B2 ✓
  • Generate multiple thoughts at each step
  • Evaluate each thought (self-evaluation or heuristic)
  • Prune unpromising branches, explore promising ones
Tree of Thoughts extends CoT from a single linear chain to a branching tree. At each reasoning step, the model generates multiple possible next thoughts. Then it evaluates which branches are promising and prunes the rest. This is essentially search (BFS or DFS) over the reasoning space, using the LLM both as the generator and the evaluator. ToT is especially powerful for planning, puzzle-solving, and creative tasks where the first approach might not be the best. The downside: it requires many more API calls (one per node in the tree). In practice, ToT is used for high-value offline tasks, not real-time applications.

ReAct: Reasoning + Acting

Yao et al., "ReAct: Synergizing Reasoning and Acting in Language Models" (2022)

Thought: I need to find the population of France and Germany to compare them.
Action: search("population of France 2025")
Observation: France population: ~68.5 million (2025 est.)
Thought: Now I need Germany's population.
Action: search("population of Germany 2025")
Observation: Germany population: ~84.5 million (2025 est.)
Thought: Germany (84.5M) has a larger population than France (68.5M) by about 16M.
Action: finish("Germany has a larger population than France: ~84.5M vs ~68.5M.")

The pattern: Thought → Action → Observation → repeat

  • Combines reasoning (CoT) with tool use
  • Foundation of AI agents (LangChain, AutoGPT, etc.)
ReAct is the bridge between prompting and AI agents. The key innovation: interleave reasoning (thinking about what to do) with acting (actually doing it). The Thought-Action-Observation loop lets the model plan, execute, observe results, and adjust. This is the foundation of every modern AI agent framework: LangChain, LlamaIndex, AutoGPT, CrewAI. When you hear "tool use" or "function calling" in LLM APIs, it's implementing ReAct under the hood. The LLM decides WHICH tool to call and with WHAT arguments, observes the result, then decides what to do next. This is extremely powerful and is the basis of what underpins retrieval and reasoning in RAG pipelines, which we'll explore in Session 2.

Prompt Chaining

Break complex tasks into a pipeline of simpler prompts.

Step 1
Extract key facts
→
Step 2
Analyze & reason
→
Step 3
Format output

Example: Code review pipeline

Prompt 1

"List all functions in this code with their signatures and purpose."

→

Prompt 2

"Given these functions, identify security issues, bugs, and anti-patterns."

→

Prompt 3

"Generate a code review report with severity ratings and fix suggestions."

Each prompt is simple and focused. The output of one becomes the input of the next.

Prompt chaining is the most practical advanced technique for production systems. Instead of writing one mega-prompt that tries to do everything, break it into steps. Each step gets a focused prompt and produces intermediate output that feeds the next step. Benefits: each prompt is simpler and more reliable, you can test and debug each step independently, and you can use different models or temperatures for different steps. For example, use a cheap fast model for extraction in step 1, and a powerful model for analysis in step 2. This is essentially software engineering applied to prompting: decomposition and modularity.

Techniques at a Glance

Technique Complexity Cost Best For
Zero-shot Low 1x Simple, well-defined tasks
Few-shot Low-Medium 1x (more tokens) Pattern matching, extraction
CoT Low 1x (more output) Math, logic, multi-step reasoning
Self-Consistency Medium 5-10x High-stakes decisions, math
Tree of Thoughts High 10-50x Planning, puzzles, creative tasks
ReAct High Variable Tasks requiring external tools/data
Prompt Chaining Medium 2-5x Complex multi-stage workflows

Start simple (zero-shot), add complexity only when needed. Simplest effective technique wins.

Here's your decision framework. Always start with zero-shot. If that doesn't work, try few-shot. If reasoning quality is the issue, add CoT. If reliability matters, use self-consistency. ToT and ReAct are for specialized use cases. The guiding principle: use the simplest technique that achieves your accuracy target. More complex techniques cost more in API calls, latency, and code complexity. Prompt chaining is often the sweet spot for production systems: moderate cost, good reliability, and easy to debug.

Part 6

Evaluation and Pitfalls

Before we wrap up, let's talk about what can go wrong. These are the failure modes every engineer needs to understand.

Pitfall 1: Hallucination

The model generates confident, fluent, but factually incorrect text.

LLMs are trained to produce plausible text, not truthful text.

Common hallucination types

  • Fabricated citations / papers
  • Incorrect API signatures
  • Made-up statistics
  • Plausible but wrong code

Mitigation strategies

  • RAG: ground responses in real documents
  • Ask for sources: "cite your references"
  • Constrain scope: "only use info from the provided text"
  • Verification: automated fact-checking pipelines
Hallucination is the number one risk when using LLMs in production. The model has no notion of truth; it generates what's statistically likely given the prompt. If you ask "list 5 papers on X", it may invent paper titles, authors, and DOIs that look real but aren't. Mitigation: always ground the model in real data when accuracy matters. RAG (Retrieval-Augmented Generation) is the standard approach: retrieve relevant documents first, then ask the model to answer based on those documents. Also: tell the model "say I don't know if you're not sure". This helps but doesn't eliminate the problem.

Pitfall 2: Sensitivity & Bias

Prompt Sensitivity

"Is this review positive?"
→ Yes (90% confidence)
"Is this review negative?"
→ Yes (75% confidence)

Same review! Different framing → different answer.

Bias in Prompting

  • Anchoring — model follows the framing you set
  • Example bias — few-shot examples skew output distribution
  • Sycophancy — model agrees with your stated opinion
  • Position bias — in multiple-choice, preference for certain positions

Best practice: Test prompts with diverse inputs. Use neutral phrasing. Balance few-shot examples across categories.

Two related issues that trip up even experienced practitioners. Prompt sensitivity: small changes in wording can flip the model's answer. "Is this positive?" and "Is this negative?" can get different answers for the same input. This is because the model is influenced by the framing of the question. Bias: LLMs inherit biases from training data, and prompts can amplify or mitigate them. Sycophancy is particularly sneaky: if you say "I think X", the model is more likely to agree. Always test prompts with diverse inputs and edge cases. Use neutral phrasing for evaluations. If you're doing classification with few-shot examples, make sure to balance examples across classes.

Your Prompting Toolkit

A universal template you can adapt for any task:

[SYSTEM] You are a {role} with expertise in {domain}. Follow these rules: {rules}.
[CONTEXT] Background: {relevant context}  ·  Constraints: {limitations and requirements}
[EXAMPLES] (optional)   Input: {example_input_1}  →  Output: {example_output_1}
[TASK] {specific instruction}
[FORMAT] Respond in the following format: {output_schema_or_template}
[REASONING] (optional) Think step by step before answering.

Not every prompt needs all sections. Start minimal, add sections when output quality is insufficient.

Here's a universal template that covers everything we discussed today. SYSTEM sets the role and rules. CONTEXT provides background. EXAMPLES teach the pattern (few-shot). TASK is the actual instruction. FORMAT specifies output structure. REASONING triggers CoT. You don't need all sections every time. Start with just TASK. If the output isn't good enough, add SYSTEM for role-setting. If the format is wrong, add FORMAT. If reasoning is off, add REASONING. Think of it as progressive enhancement: add complexity only as needed. I encourage you to save this template and adapt it for your own projects.

Key Takeaways

  1. Prompting is a new programming paradigm — natural language as code
  2. Zero-shot first, few-shot when needed — always start simple
  3. Be specific: role + context + format + constraints
  4. Chain-of-Thought unlocks multi-step reasoning ("think step by step")
  5. Advanced techniques (Self-Consistency, ToT, ReAct) trade cost for accuracy
  6. Always validate: beware of hallucination, sensitivity, and bias
